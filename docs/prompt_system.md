# FORPY — Prompt System

## Overview

FORPY's prompt system is built around two guiding principles: **uniformity** and **cost efficiency**. Every exercise generated by the app — regardless of level, sub-level, or AI provider — produces the same JSON structure. And by separating the fixed system prompt from the dynamic user request, the system is designed to take full advantage of **prompt caching**, dramatically reducing API costs for users who generate exercises in volume.

---

## Universal JSON Response Format

All 25 sub-level prompts instruct the AI to respond with the same 4-field JSON structure:

```json
{
  "enonce":      "Clear, simple statement of the exercise",
  "correction":  "Complete Python solution, ready to copy",
  "explication": "Line-by-line explanation, no jargon",
  "deroulement": "Fluent oral description of how the code runs"
}
```

This uniformity is a deliberate architectural choice. A single JSON parser handles every response across every level, every chapter, and every AI provider. Adding a new chapter (Python ML, Python AI) or a new sub-level requires writing a new prompt file — the parser, the display logic, and the exercise screen remain untouched.

### The 4 Fields Explained

| Field | Display | Purpose |
|---|---|---|
| `enonce` | 📝 Statement | The exercise brief — clear and concise, with code only when relevant (e.g. debug exercises) |
| `correction` | ✅ Correction | Complete, runnable Python code — displayed in monospace, VS Code style |
| `explication` | 💡 Explanation | Line-by-line breakdown, written without technical jargon |
| `deroulement` | 🎙 Walkthrough | A fluent, oral-style paragraph describing how the code executes — designed to be read aloud |

## Prompt Architecture — Fixed System + Dynamic Request

Each prompt file is split into two logical parts:

### 1. Fixed System Prompt (cached)

The system prompt contains everything that never changes between generations:

- The role and persona of the AI ("you are a Python teacher...")
- The pedagogical level and constraints of the sub-level
- The strict JSON output format with field definitions
- The rule to never repeat the last exercise topic
- The instruction to respond in `{language}`

This part is **identical on every call** for a given sub-level. It is the ideal candidate for prompt caching.

### 2. Dynamic User Request (never cached)

The user part contains only what changes between generations:

- The specific generation instruction ("generate a new exercise")
- The `{last_topic}` constraint ("avoid the topic: lists")
- For pattern sub-levels: `{category}` and `{pattern}` ("category: Dictionaries, pattern: .items()")

## Prompt Caching — Cost Efficiency at Scale

FORPY is designed for **high-volume exercise generation**. A user working through a level might generate 20, 50, or 100 exercises in a single session. Without caching, the full system prompt is re-tokenised and re-billed on every single call.

### How Prompt Caching Works (Anthropic Claude)

When the system prompt is marked for caching, Claude stores it in memory for **5 minutes**. Any call within that window reuses the cached version — the cached tokens are billed at **10% of the standard input price**.

```
First call:   full system prompt billed at standard rate
              └── cache is created (valid 5 minutes)

Calls 2–N:    system prompt billed at 10% of standard rate
              └── cache hit, only the dynamic part is new
```

### Cost Impact Example

A typical FORPY system prompt is ~500–800 tokens. For a user generating 50 exercises in a session:

| Scenario | Tokens billed at full price | Estimated saving |
|---|---|---|
| Without caching | 50 × ~700 = 35,000 tokens | — |
| With caching | 700 + 49 × 70 = 4,130 tokens | ~88% reduction |

For users generating exercises in volume — which is the entire point of FORPY — this saving is significant and directly reduces their API bill.

### Implementation in `ai_service.py`

Prompt caching is implemented via the `cache_control` parameter in the Anthropic API call. Only the system prompt block is marked for caching — the user message is always sent fresh.

```python
# Anthropic prompt caching — system prompt cached, user message fresh
response = client.messages.create(
    model="claude-opus-4-5",
    max_tokens=1024,
    system=[
        {
            "type": "text",
            "text": system_prompt,
            "cache_control": {"type": "ephemeral"}  # cache for 5 minutes
        }
    ],
    messages=[
        {
            "role": "user",
            "content": user_message  # dynamic part — never cached
        }
    ]
)
```

For OpenAI and Gemini providers, caching behaviour depends on the provider's implementation. The prompt structure (fixed system + dynamic user) is maintained across all providers to maximise compatibility with any future caching support.

## Variable Injection

Before sending the prompt to the AI, `prompt_builder.py` injects runtime variables into the prompt text using simple string replacement:

| Variable | Source | Purpose |
|---|---|---|
| `{language}` | `settings.json` → `exercise_language` | Language of the generated exercise (EN, FR, etc.) |
| `{last_topic}` | `session_state` → `last_exercise_topic` | Last generated topic — injected as an exclusion constraint |
| `{category}` | `session_state` → `current_pattern_category` | Pattern category selected by the user (pattern sub-levels only) |
| `{pattern}` | `session_state` → `current_pattern` | Specific pattern selected by the user (pattern sub-levels only) |

```python
def get_prompt(self, chapter, sublevel, language,
               last_topic="", category="", pattern=""):
    path = f"prompts/{chapter}/{sublevel}.txt"
    prompt = open(path).read()
    prompt = prompt.replace("{language}",   language)
    prompt = prompt.replace("{last_topic}", last_topic)
    prompt = prompt.replace("{category}",   category)
    prompt = prompt.replace("{pattern}",    pattern)
    return prompt
```

## Anti-Repetition System

Every time an exercise is generated, its topic is saved in `session_state`:

```json
{ "last_exercise_topic": "list comprehensions" }
```

On the next generation, this topic is injected into the prompt as a hard constraint:

```
Do NOT generate an exercise about: list comprehensions.
Generate an exercise on a DIFFERENT topic within this sub-level.
```

When the user navigates to a different sub-level, `last_exercise_topic` is reset to an empty string — the constraint no longer applies.

## Extensibility

The prompt system is designed to scale horizontally:

- **New chapter** (e.g. `python_ml/`): create a folder, add prompt files, register the chapter — the parser, display, and caching logic are reused as-is
- **New language**: add the language code to `translations.json` — `{language}` injection handles the rest
- **New sub-level type**: write a new prompt file following the universal JSON contract — no code changes required
- **New AI provider**: implement `AIProvider.call()` — the prompt structure is provider-agnostic
