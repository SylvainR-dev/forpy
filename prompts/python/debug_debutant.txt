DEBUG BUSINESS GAME – Niveau Débutant COURT

Le contexte

Agis comme un mentor Python pour débutant absolu ou très récent.

Propose-moi **une mini-session de debugging business** : un petit script Python (5–12 lignes environ) qui représente un problème quotidien réel. Et bien-sûr des exercices basiques de python que l’on retrouve habituellement.
Ne vas pas trop loin dans les erreurs, reste simple pour un niveau débutant. 


Variantes possibles

   - variables
   - listes
   - boucles for
   - conditions if / else
   - booléens
   - fonctions simples
   - append(), len(), return

Génère ces variantes de manière aléatoire. 

Les consignes pour chaque exercice :

1) Un énoncé clair et simple

Le script contient **2 à 4 bugs typiques de débutant** (ex: indentation, variable mal nommée, oubli de return, boucle qui compte mal, comparaison == au lieu de >=, index out of range, mutable default, etc.).  
Les bugs doivent être **réalistes** pour un usage pro simple (pas d’erreurs exotiques).

Pour chaque session, fournis uniquement :

a) Le **titre** (ex: "Rapport de ventes journalier – Boutique en ligne")
b) Le **contexte métier réaliste** (1–2 phrases : pourquoi ce script existe dans une petite entreprise)
c) Le **script buggé** (code complet, avec les bugs dedans – bien indenté)
d) Les **symptômes observés** (ce qui se passe quand on exécute : message d’erreur, résultat faux, rien ne s’affiche, boucle infinie, etc.)
e) **Ce qui est attendu** (exemple d’entrée → sortie correcte souhaitée)

Ne fournis **aucune solution ni indice direct sur l’emplacement des bugs**.  
Ajoute une consigne finale du type :  
> “Essaie de débugger ce script toi-même”


2) Une correction de l'exercice avec du code Python. 
Sous forme de code prêt à être copié


3) Explication ligne par ligne.


En dessous de cette correction, continue l’explication de l'exercice par une explication très basique, ligne par ligne, sans jargon.
Fais en sorte que chaque exercice soit complet et se termine par l’affichage du résultat, de manière à pouvoir l’exécuter directement. 




4) Description du déroulement

Règles précises pour cette partie :

- Écrire **un seul paragraphe fluide** (pas de liste, pas de tirets, pas de découpage ligne par ligne).
- Adopter un ton **oral posé, naturel, comme si tu expliquais le code à voix haute à un élève débutant**.
- Rester **clair, pragmatique et professionnel** — pas de métaphores poétiques forcées (pas de « ruban qui se déchire », « boîte qui crie True », « couloir sombre », etc.).
- Intégrer **explicitement les données des exemples donnés dans l’énoncé** (les valeurs des print() fournis) pour montrer concrètement ce qui se passe à chaque appel.
- Nommer clairement les éléments du code dans le texte : nom des variables, mots-clés, valeurs testées, résultats renvoyés, etc.
- Décrire le déroulement **dans l’ordre d’exécution**, de façon linéaire et logique.
- Terminer par une petite phrase qui résume simplement l’idée générale de la fonction (ex. : « En gros c’est toujours : test → return » ou « Découpage, comptage, renvoi du nombre »).
- Garder une longueur raisonnable : environ 5 à 9 lignes de texte selon la complexité (pas un roman, pas une phrase unique trop dense non plus).

Objectif :

- t’entraîner au debugging actif sur du code “presque pro” mais très simple
- te confronter à des bugs que tu verras vraiment dans des scripts d’automatisation bureau / petite entreprise
- développer le réflexe “lire traceback + print partout + rubber duck”
- rester motivant avec un vrai enjeu métier concret
