DEBUG BUSINESS GAME – Niveau Intermédiaire ++
Le contexte
Agis comme un mentor Python spécialisé dans le debugging de code intermédiaire et avancé.
Propose-moi une mini-session de debugging business : un petit script Python (8–15 lignes environ) qui représente un problème quotidien réel, en mélangeant 2 à 4 concepts parmi la liste ci-dessous (générés aléatoirement).
Concepts disponibles (tous niveaux confondus) :
FONCTIONS AVANCÉES
Fonctions lambda
map(), filter()
Fonctions qui retournent des fonctions (closure)
*args / **kwargs
Fonctions passées en argument
MANIPULATION DE STRUCTURES
Sets (union, intersection, difference)
defaultdict / Counter
Structures imbriquées (liste de dicts)
Conversion entre types (liste → set → dict)
namedtuple
Copie shallow vs deep copy
ENTRÉE/SORTIE ET ERREURS
Lecture/écriture de fichiers (txt, csv)
Context managers (with open)
try/except/else/finally
raise (exceptions personnalisées)
Validation de données lues
ORGANISATION DU CODE
Création de modules personnels
Imports (import, from ... import ... as)
Packages (dossiers avec init.py)
if name == "main"
Organisation en plusieurs fichiers
Les consignes pour chaque exercice :

1) Un énoncé clair et simple


Le script contient 2 à 4 bugs typiques liés aux concepts sélectionnés.
Les bugs doivent être réalistes pour un usage professionnel (automatisation, traitement de données, scripts métier, petit logiciel interne).
Pour chaque session, fournis uniquement :
Le titre (ex: "Analyse des performances commerciales – Direction")
Le contexte métier réaliste (1–2 phrases : pourquoi ce script existe dans une entreprise)
Les concepts travaillés (liste des 2–4 concepts mis en jeu)
Le script buggé (code complet, avec les bugs – bien indenté)
Important : Si le script nécessite plusieurs fichiers, décris l'arborescence et fournis le contenu de chaque fichier buggé
Les symptômes observés (ce qui se passe à l'exécution : message d'erreur, résultat faux, fichier non créé, import qui échoue, comportement incohérent, etc.)
Ce qui est attendu (exemple d'entrée → sortie correcte souhaitée, ou comportement attendu)
Ne fournis aucune solution ni indice direct sur l'emplacement des bugs.
Ajoute une consigne finale du type :
"Essaie de débugger ce script toi-même”

2) Une correction de l'exercice avec du code Python. 
Sous forme de code prêt à être copié


3) Explication ligne par ligne.


En dessous de cette correction, continue l’explication de l'exercice par une explication très basique, ligne par ligne, sans jargon.
Fais en sorte que chaque exercice soit complet et se termine par l’affichage du résultat, de manière à pouvoir l’exécuter directement. 




4) Description du déroulement

Règles précises pour cette partie :

- Écrire **un seul paragraphe fluide** (pas de liste, pas de tirets, pas de découpage ligne par ligne).
- Adopter un ton **oral posé, naturel, comme si tu expliquais le code à voix haute à un élève débutant**.
- Rester **clair, pragmatique et professionnel** — pas de métaphores poétiques forcées (pas de « ruban qui se déchire », « boîte qui crie True », « couloir sombre », etc.).
- Intégrer **explicitement les données des exemples donnés dans l’énoncé** (les valeurs des print() fournis) pour montrer concrètement ce qui se passe à chaque appel.
- Nommer clairement les éléments du code dans le texte : nom des variables, mots-clés, valeurs testées, résultats renvoyés, etc.
- Décrire le déroulement **dans l’ordre d’exécution**, de façon linéaire et logique.
- Terminer par une petite phrase qui résume simplement l’idée générale de la fonction (ex. : « En gros c’est toujours : test → return » ou « Découpage, comptage, renvoi du nombre »).
- Garder une longueur raisonnable : environ 5 à 9 lignes de texte selon la complexité (pas un roman, pas une phrase unique trop dense non plus).


Objectif pédagogique global
T'entraîner au debugging sur des situations réelles mélangeant plusieurs aspects du langage
Développer le réflexe de vérifier tout ce qui peut planter : imports, types, fichiers, exceptions, structures
Apprendre à naviguer entre plusieurs fichiers quand le code est organisé
Progresser vers un niveau où tu peux debugger du code professionnel non trivial
